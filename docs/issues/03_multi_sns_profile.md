# Issue: 自主登録フロー ＆ リアルタイム印刷管理とQRコード分割印刷

*   **Type:** Feature (仕様変更・機能拡張)
*   **Labels:** `feature` 🆕, `print-queue` 🖨️, `multi-sns` 🔗, `self-checkout` 📱
*   **Priority:** Critical

---

## 概要 (Overview)

当初計画していた「その場で主催者が名前を聞いて登録・印刷するフロー」は受付の混雑を招くため、**「参加者が自身のスマートフォンで自主登録し、主催者端末でリアルタイム自動印刷するセルフ受付フロー」**へと大きく方向転換します。

参加者は受付の案内QRコード等から自身のスマホで登録ページにアクセスし、「名前」「所属」「複数のSNS（任意）」を登録します。登録完了と同時にデータは主催者側のタブレット（プリンター接続中）にリアルタイムで転送され、サーマルプリンターから名前札が自動的に出力されます。参加者は受付テーブルに並べられた自身の名前札を探し、ネームホルダーに入れて入場します。

また、名前札に印字するQRコードはスキャンしやすさや視認性を考慮し、**「1枚の用紙につき最大2個まで」**とし、3個以上のSNSが登録された場合は自動的に用紙（画像Canvas）を2個ずつに分割出力する仕様とします。SNSが登録されなかった場合は、QRコードなしで出力します。

## ユーザー体験 (User Experience)

### 1. 一般参加者側フロー
1.  **登録画面 (`/`):**
    *   参加者はスマホで登録画面を開き、「お名前（必須）」「所属（必須）」を入力します。
    *   任意のSNS（Instagram、X、GitHub、Discord、自由リンク）を登録します。
    *   X、GitHub、Discordは、手動入力を省くために**Mock OAuth連携ボタン**を提供します。ボタンを押すと承認確認画面（ポップアップ）が表示され、アカウント名を確認・入力して承認することで、自動でアカウント名がセットされます。
2.  **登録完了画面 (`/register-success`):**
    *   登録完了のメッセージとともに、「受付テーブルに印刷された名前札が順次並びますので、ご自身のお名前のものを探して、ネームホルダーに入れてご入場ください」といった会場内の案内が表示されます。
    *   登録した内容から生成された名前札のプレビュー画像が表示されます。ライト/ダークテーマの切り替え表示、およびPNG画像としての保存も可能です。

### 2. 主催者（受付管理）側フロー
1.  **印刷キュー画面 (`/admin/queue`):**
    *   一般参加者がスマホで登録を行うと、**画面のリロードなしで即座に**キューリスト（登録者一覧）に新しい行が追加されます。
    *   画面上部には「総登録者数」「印刷待ち」「印刷完了」の統計カウンターが配置されます。
    *   **「自動印刷モード (Auto-Print)」**トグルがONの場合、新規登録が追加された瞬間に、自動でプリンター（擬似）に印刷コマンドが送信され、印刷ステータスが「印刷完了」に移行します。
    *   画面右側には「プリンターConsoleログ」が設置され、USB接続状態やデータのスプール状況、用紙の分割処理などがモノスペースフォントでリアルタイムに流れるように出力されます。
    *   キューの各行には「印刷する/再印刷」ボタンがあり、手動での印刷実行（PNG画像ダウンロード）が可能です。行を選択すると、右側のサイドバーにその参加者の名前札（分割分を含むすべて）のプレビューがその場で表示されます。

## 実装要件 (Requirements)

### 1. 印刷レイアウト & QRコード分割ロジック (`src/app/utils/passGenerator.ts`)
*   **SNS未登録の場合:**
    *   QRコードは一切表示せず、お名前と所属のみをカード中央に大きく配置した「名前札のみ」のデザインで1枚出力します。
*   **SNS登録数が1〜2個の場合:**
    *   名前・所属・SNS情報を配置した名前札を1枚出力します。
    *   QRコードは最大2個まで横並びで配置します。
*   **SNS登録数が3個以上の場合:**
    *   1枚の用紙サイズ（600x380px）の制限内に綺麗に収めるため、**「1枚につき最大2個のQRコード」**となるように印刷データを分割します。
    *   例: 3つのSNS（Instagram、X、GitHub）を登録した場合：
        *   カード1: 名前・所属 ＋ Instagram・XのQRコード（計2個）
        *   カード2: 名前・所属 ＋ GitHubのQRコード（計1個）
        *   合計2枚の画像として出力します。
    *   複数枚に分割されたカードの右上には、受け取りミスや紛失を防ぐために「CARD 1 / 2」「CARD 2 / 2」のような**ページ数インジケーター**を描画します。

### 2. データモデルと同期制御 (`src/app/utils/types.ts`, `storage.ts`)
*   データモデル `Registration` は以下の構造とします。
    *   `id`: 一意のID文字列
    *   `name`: お名前
    *   `affiliation`: 所属
    *   `snsUrls`: 各SNS（instagram, x, github, discord, custom）のURL辞書
    *   `checkedInAt`: 登録日時（ISO8601）
    *   `printStatus`: 印刷状態（`pending` | `printed`）
*   データベースやバックエンドを使用しないローカルテスト環境のため、データの共有には `localStorage` を使用します。
*   タブ間での同期には、ブラウザ標準の `storage` イベントを購読する React Hook を用い、登録画面での送信を別タブの管理画面が瞬時に検知できるようにします。

### 3. Mock OAuth承認ダイアログ (`src/app/page.tsx`)
*   X, GitHub, Discord はAPI認証のモックとして、フォーム内に「連携する」ボタンを表示します。
*   ボタン押下時に、アプリケーション名、要求する権限、アカウント名入力フォームを備えた専用のモーダルを表示します。
*   「連携を承認する」を押すと、ローディングアニメーションを数秒挟んだのち、指定されたアカウント情報を用いて正式にSNS URLをフォーム内に自動代入し、連携状態とします。

## 開発・デバッグ時のヒント (Technical Hints)

*   **HTML Canvas の非同期描画:** Canvas に QRコード（`qrcode` ライブラリのDataURL）やアイコンをレンダリングする際、画像のロードは非同期（`img.onload`）で行う必要があります。Promise を用いてすべての画像がロード完了してから Canvas データを返す処理を構築してください。
*   **Next.js ビルド互換性:** `useSearchParams` などのクライアントサイドでのみ動作する Next.js のAPIを使用するページコンポーネント（特に完了画面など）は、Next.jsのビルドエラー（SSR時のエラー）を回避するため、必ず `<Suspense>` コンポーネントでラップした子コンポーネントとして実装します。
